# Parser

The `parser/` directory contains PocScript's syntax analyzer and AST structure. It receives the token list generated by the lexer and produces a tree that represents the structure of the program.

## Files

- `parser.h` and `parser.c`: recursive descent parser and the `parserParseProgram` entry point
- `ast.h` and `ast.c`: node types, utility functions, textual serialization, and memory cleanup

## General Structure

The parser uses a simple `Parser` struct:

- `tokens`: head of the linked list
- `current`: current token during parsing

The main flow is:

1. `parserInit` receives the token list.
2. `parserParseProgram` creates an `AST_PROGRAM` node.
3. Each program item is parsed by recursive functions such as `parseStatement`, `parseAssign`, `parseLogical`, `parseType`, and `parseFactor`.
4. On error, the parser aborts with a message containing line and column information.

The current parser is strictly syntactic. It builds the tree structure, but it does not perform type checking or semantic resolution.

## Main AST Nodes

The node types defined in `ast.h` cover four main groups.

### Program structure

- `AST_PROGRAM`: file root
- `AST_BLOCK`: block delimited by `{}`
- `AST_EXPR_STMT`: expression used as a statement

### Declarations and control flow

- `AST_VAR_DECL`: variable declaration
- `AST_FUNC_DECL`: function declaration
- `AST_PARAM`: function parameter
- `AST_IF`: `if` with support for `else` and `else if`
- `AST_WHILE`: `while` loop
- `AST_FOR`: `for` loop
- `AST_RETURN`: `ret`

### Expressions

- `AST_ASSIGN`: assignments `=`, `+=`, `-=`
- `AST_BINARY`: binary operations
- `AST_UNARY`: unary operations, currently used for `!`
- `AST_CALL`: function call
- `AST_ARRAY_ACCESS`: indexed access, including multiple indices

### Literals, identifiers, and types

- `AST_ARRAY_LITERAL`: array literal, including nested arrays
- `AST_IDENTIFIER`: identifier
- `AST_INT_LITERAL`: integer
- `AST_FLOAT_LITERAL`: float
- `AST_STRING_LITERAL`: string or char literal
- `AST_BOOL_LITERAL`: boolean
- `AST_TYPE_NAME`: nominal type, builtin or custom
- `AST_TYPE_ARRAY`: array type with element type and optional size

## General AST Format

Each `AstNode` contains:

- `type`: the node kind
- `data`: a `union` with the payload for each variant

Some important shapes are:

- `AST_PROGRAM` and `AST_BLOCK`: store `items[]` lists
- `AST_VAR_DECL`: `name`, `declared_type`, `initializer`
- `AST_FUNC_DECL`: `name`, `params[]`, `body`, `return_type`
- `AST_IF`: `condition`, `then_branch`, `else_branch`
- `AST_FOR`: `init`, `condition`, `update`, `body`
- `AST_ASSIGN`: `target`, `value`, `op`
- `AST_BINARY`: `left`, `right`, `op`
- `AST_UNARY`: `operand`, `op`
- `AST_CALL`: `callee`, `args[]`
- `AST_ARRAY_ACCESS`: `base`, `indices[]`
- `AST_TYPE_ARRAY`: `element_type`, `size_expr`

## Conceptual Example

For a snippet like:

```text
x::int = 1 + 2;
```

The generated tree follows this idea:

```text
PROGRAM
└─ VAR_DECL x
   ├─ TYPE int
   └─ BINARY +
      ├─ INT_LITERAL 1
      └─ INT_LITERAL 2
```

The real textual format is produced by `astToString` and is the same one used by the integration tests.

## Internal Types And Operators

### `AstTypeKind`

- `AST_TYPE_INT_KIND`
- `AST_TYPE_FLOAT_KIND`
- `AST_TYPE_CHAR_KIND`
- `AST_TYPE_BOOL_KIND`
- `AST_TYPE_ARRAY_KIND`
- `AST_TYPE_CUSTOM_KIND`

### `AstBinaryOp`

- arithmetic: `ADD`, `SUB`, `MUL`, `DIV`
- comparison: `GT`, `LT`, `GTE`, `LTE`, `EQ`, `NEQ`
- logical: `AND`, `OR`

### `AstAssignOp`

- `AST_ASSIGN_SET`
- `AST_ASSIGN_ADD`
- `AST_ASSIGN_SUB`

## Useful API

- `parserParseProgram(...)`: parser entry point
- `astToString(...)`: serializes the tree to text
- `astPrintPretty(...)`: prints the AST to stdout
- `astFree(...)`: frees the entire tree

## Notes

- The parser assumes the lexer has already removed comments and whitespace.
- Errors are reported based on the current token, reusing `line` and `column` from the lexer.
- The textual AST serialization is part of the project's practical contract because it is used to validate parsing in the tests.

# Semantic Analyzer

The `semantic/` directory contains PocScript's semantic analysis phase. It receives the AST generated by the parser and validates declarations, name resolution, scope rules, type compatibility, function calls, function returns, and array access.

## Files

- `semantic.h` and `semantic.c`: semantic analysis entry point and the two-pass traversal
- `scope.h` and `scope.c`: scope chain and symbol table implementation
- `types.h` and `types.c`: semantic type representation and type comparison helpers
- `errors.h` and `errors.c`: structured semantic error collection

## General Behavior

The semantic analyzer is intentionally separate from the parser.

The current flow is:

1. The lexer generates a token list.
2. The parser consumes that list and builds an AST.
3. `semanticAnalyze(...)` walks the AST in two passes.
4. The result is a `SemanticResult` with collected semantic errors.

This phase does not try to recover syntax. It assumes the AST is already valid from a syntactic point of view.

## High-Level Design

The analyzer uses two passes over the AST.

### Pass 1: symbol collection

The first pass records declarations into scopes.

It is responsible for:

- creating the global scope
- registering variable declarations in the current scope
- registering function declarations in the global scope
- registering parameters in the function scope
- reporting duplicate declarations in the same scope

The output of this pass is the symbol environment needed by later validation.

### Pass 2: validation

The second pass validates how the program uses the collected symbols.

It is responsible for:

- resolving variable names
- resolving function calls
- checking initializer and assignment compatibility
- checking expression operand types
- checking boolean conditions in control flow
- checking function argument count and argument types
- checking function return presence and return type compatibility
- checking array indexing rules
- accumulating multiple errors instead of aborting at the first failure

The validation pass rebuilds local scope state as it traverses blocks and functions, while reusing globally collected function symbols as the base environment.

## Main Data Structures

### `SemanticResult`

Defined in `semantic.h`.

It contains:

- `global_scope`: the root scope built during symbol collection
- `errors`: the list of semantic errors found during analysis

### `Scope`

Defined in `scope.h`.

Each `Scope` represents one lexical scope and contains:

- `parent`: pointer to the enclosing scope
- `buckets`: hash table buckets for symbols declared in that scope
- `bucket_count`: hash table size

Conceptually, scopes form a linked chain:

```text
current scope -> parent scope -> global scope -> NULL
```

This makes entering and leaving scopes behave like a stack, while name lookup walks upward through the parent chain.

### `Symbol`

Also defined in `scope.h`.

Each symbol stores:

- `name`
- `kind`: variable or function
- `type`: semantic type for the symbol itself
- `params[]` and `param_count`: function parameter types when the symbol is a function
- for function symbols, `type` stores the declared return type
- `next`: linked-list chaining inside a hash bucket

### `SemanticType`

Defined in `types.h`.

This is the internal type representation used by semantic validation.

Current kinds include:

- `SEM_TYPE_INT`
- `SEM_TYPE_FLOAT`
- `SEM_TYPE_CHAR`
- `SEM_TYPE_BOOL`
- `SEM_TYPE_VOID`
- `SEM_TYPE_ARRAY`
- `SEM_TYPE_ERROR`

Arrays are recursive: an array type stores an `element_type` pointer to another `SemanticType`.

### `SemanticError` and `SemanticErrorList`

Defined in `errors.h`.

Errors are stored as structured data rather than pre-rendered output strings.

Each error stores:

- `kind`: semantic category such as type, name, or declaration
- `line`
- `column`
- `message`

This keeps error collection separate from presentation. The analyzer records facts, and the caller decides how to print them.

## Symbol Table Strategy

The symbol table is implemented as a hash table per scope.

Important operations are:

- `scopeDeclare(...)`: insert a symbol in the current scope only
- `scopeLookupCurrent(...)`: search only the current scope
- `scopeLookup(...)`: search current scope and then walk through parents

This supports the two key rules of lexical scoping:

- duplicate declarations are detected locally in the current scope
- name resolution can find declarations in enclosing scopes

## Type Checking Strategy

The analyzer converts AST type nodes into `SemanticType` values and then compares them structurally.

Helpers in `types.c` are responsible for:

- building a semantic type from AST type nodes
- cloning and freeing semantic types
- checking equality between types
- converting a semantic type into a short printable name

The current implementation keeps type rules intentionally strict:

- no implicit numeric promotion between `int` and `float`
- array element types must match
- function arguments must match declared parameter types
- function return expressions must match the declared return type
- conditions must resolve to `bool`

## Error Handling Strategy

The analyzer does not stop at the first error.

Instead, it appends a structured `SemanticError` to the result and keeps walking when possible. To make that work, invalid expressions usually resolve to `SEM_TYPE_ERROR`, which acts as a sentinel type that lets the traversal continue without crashing the rest of the pass.

This is why one program can produce multiple semantic diagnostics in a single run.

## Position Information

Each AST node stores `line` and `column`, copied from the relevant parser token. The semantic analyzer reuses that information when recording errors.

This avoids coupling semantic analysis to live lexer tokens while still producing useful diagnostics.

## Current Validation Coverage

The current semantic analyzer validates:

- duplicate variable declarations in the same scope
- duplicate function declarations in the global scope
- duplicate parameter names in the same function
- valid shadowing in nested scopes
- use of undeclared variables
- calls to undeclared functions
- initializer and assignment type compatibility
- arithmetic, comparison, and logical operand rules
- boolean conditions in `if`, `while`, and `for`
- argument count and argument type compatibility in function calls
- non-`void` functions containing at least one return statement
- return expression compatibility with the declared function return type
- rejection of value returns inside `void` functions
- rejection of `ret` outside a function
- array literal element compatibility
- integer-only array indexing
- rejection of indexing non-array values

## Output Responsibility

`semanticAnalyze(...)` does not print anything by itself.

It only returns structured data. The caller, such as `main.c`, is responsible for iterating through the collected errors and rendering them for the user.

## Notes

- The parser remains purely syntactic; semantic rules live entirely in this module.
- The semantic analyzer depends on AST node positions, not on token pointers.
- The current implementation favors clarity and explicit rules over aggressive inference or coercion.
